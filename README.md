# **Care Compliance Copilot**

訪問介護事業所における運営指導前の書類確認業務を支援する、コンプライアンス確認AI支援MVPです。

本プロジェクトでは、利用者情報・ケアプラン・日々の介護記録・モニタリング記録・書類ステータスをCSVで読み込み、未作成・未確認・未承認の書類や記録を検知します。  
さらに、必要に応じてAI下書き候補を生成し、職員が確認・修正したうえで「職員確認済み」として保存する業務フローを検証しています。

## **背景**

訪問介護事業所では、運営指導前に以下のような書類確認が必要になります。

* 訪問介護計画書が有効期間内に作成されているか  
* 月間モニタリング記録が作成・確認されているか  
* 日々の介護記録に必要な情報が記載されているか  
* 書類が未作成・未確認・未承認のまま残っていないか  
* 記録内容をもとに、モニタリング文書や特記事項を適切に補足できるか

しかし、これらを人手で確認する場合、確認漏れや対応遅れが発生しやすく、サービス提供責任者や管理者の負担も大きくなります。

本MVPでは、こうした課題に対して、CSVデータをもとに書類状態を検知し、必要な文書作成をAI下書き候補として支援する流れを検証しました。

## **このMVPで検証したこと**

本MVPでは、以下の一連の業務フローを検証しています。

1. 利用者情報・ケアプラン・日々の介護記録・モニタリング記録・書類ステータスをCSVで読み込む  
2. 未作成・未確認・未承認の書類や記録を検知する  
3. 利用者ごとに状態別アラートを表示する  
4. 必要に応じて、特記事項補填案や月間モニタリング下書きを生成する  
5. AI出力を確定文書ではなく、職員確認前提の下書きとして扱う  
6. 職員が内容を確認し、確認者・確認日時・確認済みステータスを保存する  
7. 最終チェックにより、主要フローが正常に完了していることを確認する

## **主な機能**

### **1\. 書類・記録アラート検知**

`document_status.csv`、`daily_records.csv`、`monitoring_records.csv` をもとに、以下のような状態を検知します。

* 必須書類が未作成である  
* 書類が未承認である  
* 当月モニタリング記録に確認不足がある  
* 日々の介護記録の特記事項が不足している  
* モニタリング記録の必須項目が不足している

### **2\. 状態別アラート表示**

以前のスコア方式ではなく、現場職員が次に確認すべき内容を直感的に把握できるよう、以下のように状態別に表示します。

* 未作成：対象書類名と件数  
* 未承認：対象書類名と件数  
* 未確認・確認不足：対象書類名と件数

例：

利用者D（U004）には、対応が必要な書類・記録が4件あります。  
未作成：0件（なし）。  
未承認：0件（なし）。  
未確認・確認不足：4件（日々の介護記録（4件））。  
職員による確認・修正・保存を行ってください。

### **3\. 特記事項補填案の生成**

日々の介護記録において、特記事項が短い記録を検知し、既存の記録項目をもとにAI補填案を作成します。

AIが参照する情報は、以下に限定しています。

* 現在の特記事項  
* 利用者の発言  
* 観察事項  
* 数値情報  
* 今後の対応

存在しない事実や医学的判断をAIが推測して追加しないよう、プロンプト内で制約を明記しています。

### 

### **4\. 月間モニタリング下書きの生成**

利用者ごとのケアプランと日々の介護記録をもとに、月間モニタリング文書の下書きを生成します。

生成内容には、以下を含めています。

* 総合評価  
* ケアプラン目標に対する評価  
* 状態変化と根拠  
* 今後の対応案  
* 職員が確認すべき事項  
* 情報不足の項目

AI下書きは確定文書ではなく、職員が確認・修正するための補助案として扱います。

### **5\. 職員確認済み保存**

AIが生成した下書きや補填案について、職員が確認したことを記録します。

保存する情報は以下です。

* AI下書き本文  
* 確認ステータス  
* 確認者  
* 確認日時

これにより、AIが自動で書類を確定するのではなく、最終判断を人間が行う運用フローを検証しています。

### **入力データ**

| ファイル名 | 内容 |
| ----- | ----- |
| `users.csv` | 利用者情報 |
| `care_plans.csv` | ケアプラン情報（`service_content`に加え、`visit_days`・`time_slot`・`service_category`で曜日・時間帯・サービス区分の突合に対応） |

| `monitoring_records.csv` | 月間モニタリング記録 |
| `document_status.csv` | 書類ステータス（人員・勤務体制／利用者・契約／サービス提供／運営・その他の4分野・15書類種別のサンプルデータを収録） |

`document_status.csv`の4分野15書類カタログのうち、勤務形態一覧表・出勤簿・資格証・研修記録は職員（`user_id`列に`職員A`等）、感染症対策マニュアル等は事業所全体（`user_id`列に`ORG`）に紐づくサンプル行として表現しています。担当者会議記録・研修記録・苦情処理記録はカタログ上「AI補填対象」に分類していますが、本MVPでは中身データを保持していないため、書類詳細画面ではステータス確認のみとしています。

### **出力データ**

Notebook実行後、または `app.py`（Streamlit版）の操作後に、`outputs/` 以下に以下のCSVが生成されます。

| ファイル名 | 内容 |
| ----- | ----- |
| `alerts_integrated.csv` | 統合アラート一覧 |

| `daily_records_note_reviewed.csv` | 職員確認済みの日々の記録 |
| `monitoring_records_with_ai_draft.csv` | 月間モニタリングAI下書き入り記録 |

## **実行方法**

### **Streamlitアプリ（app.py）**

ロジック検証済みのStreamlit版アプリをローカルで起動できます。

```
pip install -r requirements.txt
streamlit run app.py
```

起動後、サイドバーのメニューから以下を確認できます。書類を中心にした一本道の業務フロー（概要ダッシュボード→書類アラート一覧→書類詳細・AI補填プレビュー→確認キュー→確認履歴・監査ログ）を主要導線とし、個別確認用の補助画面をその下に配置しています。

**主要導線**

1. 使い方・業務フロー（業務フロー説明。入力CSVの直接確認はここのexpanderから）  
2. 概要ダッシュボード（アラート件数・分野別・重要度別などの動的集計）  
3. 書類アラート一覧（4分野15書類カタログに基づくアラート一覧。初期表示は未対応のみで、「確認済みも表示する」チェックボックスで切り替え可能。分野・種別・重要度で絞り込み、「▶ 詳細を見る」で次画面へ）  
4. 書類詳細・AI補填プレビュー（書類全体図・AI補填後プレビュー・計画書連動レ点チェック・根拠確認・職員確認保存を1画面に集約。保存後は「次の未対応書類へ進む」で連続処理可能）  
5. 確認キュー（未確認AI下書き・未承認記録・要確認レ点チェックを横断一覧化し、「詳細へ」で書類詳細画面に遷移する入口画面）  
6. 確認履歴・監査ログ（職員確認済みの履歴一覧。誰が・いつ・何を確認したかを記録）

職員が確認・保存した書類は、データを削除するのではなく`alert_key`（source_type＋対象ID＋対象＋書類名＋アラート種別から作る安定キー）で確認済みとして記録します。これにより、確認済みの書類は「書類アラート一覧」「確認キュー」の初期表示（未対応のみ）からは除外されますが、データそのものは残り続け、「確認履歴・監査ログ」では常に確認できます。アプリを再起動しても、`outputs/confirmation_queue_log.csv`と各reviewed CSVに記録が残るため、除外状態は維持されます。

**補助画面（個別に直接確認したい場合）**

7. アラート根拠確認（アラート単位で元データを直接確認）  
8. 計画書連動レ点候補（レ点候補だけを一覧表示、ルールベース）  
9. 特記事項AI補填デモ（日々の記録の特記事項AI下書きのみ）  
10. 月間モニタリングAI下書き（月間モニタリングのAI下書きのみ）  
11. 職員確認済み保存（AI下書きの確認・保存を単独画面で実施）

いずれの画面でも、AIやルールは記録・書類・レ点チェックを確定しません。下書き・候補・不一致検知を提示するのみで、最終確定は必ず職員が確認・修正・保存します。

`outputs/` はアプリ実行のたびに更新される作業ディレクトリです。出力イメージのみ確認したい場合は `sample_outputs/` を参照してください。

## **設計上のポイント**

### **AIは確定文書を作成しない**

本MVPでは、AIの出力はあくまで下書き候補として扱います。  
AIが自動で正式記録を確定するのではなく、職員が確認・修正したうえで保存する設計にしています。

### **現場職員が次に確認すべき内容を明確にする**

優先度を0〜100点で表示するのではなく、未作成・未承認・未確認の状態別に対象書類名を表示します。  
これにより、職員が「何を確認すべきか」を直感的に把握できるようにしています。

### **記録に存在しない事実をAIが追加しない**

AI下書き生成では、日々の記録やケアプランに存在する情報のみを使用するルールを明記しています。  
存在しない発言・症状・支援内容・医学的判断を推測して追加しないことを重視しています。

## **今後の拡張案**

Colabで検証したロジックは、`app.py` としてStreamlitに移植済みです。現状のStreamlit版は、書類を中心にした一本道の業務フロー（概要ダッシュボード→書類アラート一覧→書類詳細・AI補填プレビュー→確認キュー→確認履歴・監査ログ）を主要導線として、以下を操作できます。

* 4分野15書類カタログに基づく書類アラートの一覧・絞り込み  
* 書類詳細・AI補填プレビュー画面での、書類全体図・AI補填後プレビュー・計画書連動レ点チェック・根拠確認・職員確認保存の一括操作  
* 保存後に連続して次の未対応書類へ進む導線  
* 確認キューによる、未確認AI下書き・未承認記録・要確認レ点チェックの横断的な処理入口  
* 職員確認済みとしての保存（確認者・確認日時の記録、一意キーによる確認済み判定）  
* 確認履歴・監査ログの一覧確認

今後の拡張候補としては、以下を想定しています。

* 同一record\_id / monitoring\_idの重複チェックまたは上書き保存  
* 訪問介護計画書PDFの自動読み取り（現状は`care_plans.csv`を構造化済みデータとして扱っている）  
* 担当者会議記録・研修記録・苦情処理記録の中身データモデル化とAI補填の実装  
* 計画書連動レ点候補のマッチング精度向上（表記ゆれ吸収など）
